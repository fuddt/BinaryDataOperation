# Phase 3: レイアウトとは何か

struct とはメモリレイアウトの設計図である。本章ではこれを理解することが、Partial Binary Patch 設計の入口となる。

## Phase 2 の復習

- メモリは連続したバイト列
- `uint8_t*` はバイト単位ポインタ
- `buffer + offset` は offset バイト先を指す

> struct = メモリレイアウトの設計図

---

## Phase3のゴール

この章で理解すべき概念：

① structはメモリの並び方を定義する  
② structの中身は連続メモリに配置される  
③ フィールドにはoffsetがある  
④ paddingが発生する理由  
⑤ `sizeof(struct)` が単純加算にならない理由  

---

## structとは何か（本質）

まず重要な定義：

> structは「メモリ配置の定義」

例えば：

```cpp
struct A
{
    uint8_t x;
    uint8_t y;
    uint8_t z;
};
```

これはメモリ上で：

```
offset 0 → x
offset 1 → y
offset 2 → z
```

となる。

```text
struct = layout
```

---

## 実験1：sizeof(struct)

コード：

```cpp
#include <iostream>
#include <cstdint>

struct A
{
    uint8_t x;
    uint8_t y;
    uint8_t z;
};

int main()
{
    std::cout << sizeof(A) << std::endl;
}
```

予想：

```
3
```

---

## structは「連続メモリ」

```cpp
#include <iostream>
#include <cstdint>

struct A
{
    uint8_t x;
    uint8_t y;
    uint8_t z;
};

int main()
{
    A a;

    std::cout << (void*)&a.x << std::endl;
    std::cout << (void*)&a.y << std::endl;
    std::cout << (void*)&a.z << std::endl;
}
```

出力：

```
0x1000
0x1001
0x1002
```

```text
struct = 連続メモリ
```

---

## paddingが発生する

```cpp
struct B
{
    uint8_t a;
    int b;
};
```

直感：

```
1 + 4 = 5バイト
```

でも実際の `sizeof(B)` は：

```
8
```

になることが多い。理由は **padding** にある。

---

## paddingとは何か

CPUは4バイト境界・8バイト境界でアクセスすると効率が良い。そのため：

```cpp
uint8_t a;
int b;
```

はこう配置される：

```
offset 0 → a
offset 1
offset 2
offset 3 ← padding
offset 4 → b
```

合計8バイトになる。

---

## offsetofを使う

struct のフィールドが何バイト目にあるかを確認するマクロ。

コード：

```cpp
#include <iostream>
#include <cstddef>

struct B
{
    uint8_t a;
    int b;
};

int main()
{
    std::cout << offsetof(B, a) << std::endl;
    std::cout << offsetof(B, b) << std::endl;
}
```

結果：

```
0
4
```

`b` は offset 4 にあることが確認できる。

---

## structは「レイアウト表」

```cpp
struct Horse
{
    uint8_t birthday[8];
    uint8_t owner[16];
    uint8_t jockey[12];
};
```

これはつまり：

```
offset 0   birthday
offset 8   owner
offset 24  jockey
```

という意味になる。

```text
struct = layout descriptor
```

---

## structとbufferの関係

- buffer：意味なし
- struct：意味付き

```text
buffer + struct = 解釈
```

---

## 実験：bufferをstructとして読む

コード：

```cpp
#include <iostream>
#include <cstdint>

struct A
{
    uint8_t x;
    uint8_t y;
    uint8_t z;
};

int main()
{
    uint8_t buffer[3] = {10,20,30};

    A* a = reinterpret_cast<A*>(buffer);

    std::cout << (int)a->x << std::endl;
    std::cout << (int)a->y << std::endl;
    std::cout << (int)a->z << std::endl;
}
```

結果：

```
10
20
30
```

bufferをstructとして読み取ることができる。

---

## reinterpret_cast の安全性

この操作が安全なのは、**layout が一致している場合のみ**となる。

---

## layoutが複数ある場合の問題

想定するケース：

```
layoutA
layoutB
layoutC
```

のように複数のフォーマットが存在する場合、`struct Horse` を1種類だけ定義すると、フォーマットによっては壊れる。そのため **offset操作** による柔軟な設計が必要になる。

④ `offsetof` が使える  
⑤ struct = layout descriptor
