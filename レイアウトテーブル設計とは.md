では **Phase4：layoutとは何か** に入る。

ここから、あなたの現場問題にかなり近づく。

---

# Phase4のゴール

最終的に理解することはこれ：

> layout = 「何バイト目から、何バイト分が、何の意味を持つか」を定義した表

つまり、bufferという「意味のない板」に対して、意味を与えるための地図。

---

# 1. bufferだけでは意味がない

例えば：

```cpp
uint8_t buffer[12] = {
    20, 24, 4, 25,
    1, 2, 3, 4,
    80, 81, 82, 83
};
```

これはただの12バイト。

このままだと：

```text
20 24 4 25 1 2 3 4 80 81 82 83
```

にしか見えない。

でもlayoutがあると意味が出る。

---

# 2. layoutがあると意味が決まる

例えばlayoutがこうだとする：

```text
offset 0, size 4  = birthday
offset 4, size 4  = owner
offset 8, size 4  = jockey
```

すると同じbufferがこう見える：

```text
birthday = 20 24 4 25
owner    = 1 2 3 4
jockey   = 80 81 82 83
```

つまり：

```text
buffer + layout = 意味のあるデータ
```

---

# 3. structもlayoutの一種

Phase3で見たこれ：

```cpp
struct HorseRecord
{
    uint8_t birthday[4];
    uint8_t owner[4];
    uint8_t jockey[4];
};
```

これは実質こういうlayout：

```text
birthday offset 0 size 4
owner    offset 4 size 4
jockey   offset 8 size 4
```

つまり：

```text
struct = C++コードで書いたlayout
```

---

# 4. でもstruct layoutには弱点がある

structは固定レイアウト。

例えばFormat A：

```text
birthday offset 0 size 4
owner    offset 4 size 4
jockey   offset 8 size 4
```

Format B：

```text
birthday offset 0 size 4
jockey   offset 4 size 4
owner    offset 8 size 4
```

この2つは配置が違う。

Format A用struct：

```cpp
struct HorseRecordA
{
    uint8_t birthday[4];
    uint8_t owner[4];
    uint8_t jockey[4];
};
```

Format B用struct：

```cpp
struct HorseRecordB
{
    uint8_t birthday[4];
    uint8_t jockey[4];
    uint8_t owner[4];
};
```

こうやってformatごとにstructを増やすことはできる。

でも、差し替え処理までformatごとに分岐すると地獄になる。

---

# 5. だからlayout tableにする

structで固定する代わりに、layoutをデータとして持つ。

```cpp
#include <cstddef>

struct FieldDescriptor
{
    size_t offset;  // bufferの先頭から何バイト目か
    size_t size;    // 何バイト分か
};
```

これがlayout tableの最小単位。

---

# 6. field名も必要になる

どのフィールドを書き換えるかを識別するために、IDを作る。

```cpp
enum class FieldID
{
    Birthday,
    Owner,
    Jockey
};
```

そしてlayoutをこう持つ：

```cpp
#include <unordered_map>

using LayoutTable = std::unordered_map<FieldID, FieldDescriptor>;
```

---

# 7. Format Aのlayout

```cpp
LayoutTable layoutA = {
    { FieldID::Birthday, {0, 4} },
    { FieldID::Owner,    {4, 4} },
    { FieldID::Jockey,   {8, 4} }
};
```

---

# 8. Format Bのlayout

```cpp
LayoutTable layoutB = {
    { FieldID::Birthday, {0, 4} },
    { FieldID::Jockey,   {4, 4} },
    { FieldID::Owner,    {8, 4} }
};
```

ここが重要。

**Ownerの意味は同じでも、offsetが違う。**

だからlayout tableが効く。

---

# 9. layout tableを使って読み取る

```cpp
#include <iostream>
#include <cstdint>
#include <cstring>
#include <unordered_map>

enum class FieldID
{
    Birthday,
    Owner,
    Jockey
};

struct FieldDescriptor
{
    size_t offset;
    size_t size;
};

using LayoutTable = std::unordered_map<FieldID, FieldDescriptor>;

// bufferから指定フィールドを読み取って表示する
void printField(const uint8_t* buffer,
                const LayoutTable& layout,
                FieldID field)
{
    auto it = layout.find(field);

    if (it == layout.end())
    {
        std::cout << "field not found\n";
        return;
    }

    const FieldDescriptor& desc = it->second;

    for (size_t i = 0; i < desc.size; i++)
    {
        std::cout << (int)buffer[desc.offset + i] << " ";
    }

    std::cout << std::endl;
}

int main()
{
    uint8_t buffer[12] = {
        20, 24, 4, 25,
        1, 2, 3, 4,
        80, 81, 82, 83
    };

    LayoutTable layoutA = {
        { FieldID::Birthday, {0, 4} },
        { FieldID::Owner,    {4, 4} },
        { FieldID::Jockey,   {8, 4} }
    };

    printField(buffer, layoutA, FieldID::Owner);
}
```

出力：

```text
1 2 3 4
```

# 10. layout tableを使って書き換える

これがpartial binary patchの手前。

```cpp
#include <iostream>
#include <cstdint>
#include <cstring>
#include <unordered_map>

enum class FieldID
{
    Birthday,
    Owner,
    Jockey
};

struct FieldDescriptor
{
    size_t offset;
    size_t size;
};

using LayoutTable = std::unordered_map<FieldID, FieldDescriptor>;

// 指定フィールドだけを書き換える
bool patchField(uint8_t* buffer,
                size_t bufferSize,
                const LayoutTable& layout,
                FieldID field,
                const uint8_t* newData,
                size_t newDataSize)
{
    auto it = layout.find(field);

    if (it == layout.end())
    {
        return false;
    }

    const FieldDescriptor& desc = it->second;

    // 書き換え先がbuffer範囲外なら危険なので拒否
    if (desc.offset + desc.size > bufferSize)
    {
        return false;
    }

    // サイズが一致しないなら拒否
    // 固定長バイナリではここを曖昧にすると壊れやすい
    if (newDataSize != desc.size)
    {
        return false;
    }

    // bufferの該当範囲だけを書き換える
    std::memcpy(buffer + desc.offset, newData, desc.size);

    return true;
}

int main()
{
    uint8_t buffer[12] = {
        20, 24, 4, 25,
        1, 2, 3, 4,
        80, 81, 82, 83
    };

    LayoutTable layoutA = {
        { FieldID::Birthday, {0, 4} },
        { FieldID::Owner,    {4, 4} },
        { FieldID::Jockey,   {8, 4} }
    };

    uint8_t newOwner[4] = {9, 9, 9, 9};

    bool ok = patchField(buffer,
                         sizeof(buffer),
                         layoutA,
                         FieldID::Owner,
                         newOwner,
                         sizeof(newOwner));

    std::cout << "patch result: " << ok << std::endl;

    for (uint8_t b : buffer)
    {
        std::cout << (int)b << " ";
    }

    std::cout << std::endl;
}
```

出力：

```text
patch result: 1
20 24 4 25 9 9 9 9 80 81 82 83
```

ここで起きたこと：

```text
offset 4〜7 だけを書き換えた
```

birthdayとjockeyは触っていない。

---

# 11. 今回の現場問題に戻す

あなたが最初に言っていた問題は、たぶんこれ：

```text
Format Aでは owner は offset 4
Format Bでは owner は offset 8
でも「ownerを書き換える」という処理は共通化したい
```

そのためには：

```text
処理をformatごとに変える
```

のではなく、

```text
layoutだけformatごとに変える
```

のが良い。

つまり：

```text
patch処理は1個
layoutはformatごとに複数
```

これが設計の核。

---

# Phase4の結論

Phase4で覚えるべき式はこれ：

```text
buffer = 生データ
layout = 意味の地図
field = 書き換え対象
patch = offset + size を使った部分更新
```

そして一番重要なのはこれ：

```text
struct中心ではなく、layout中心に考える
```

次のPhase5では、これを本格的に **partial binary patch設計** にする。
